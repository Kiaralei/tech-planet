## 📚 前言

先抛一个 AI 工程里几乎绕不开的矛盾：

> 你希望 Agent **能力越多越好**，但能力越多，要塞给模型的说明书就越长，上下文就越贵、越挤、越乱。

把所有工具、所有规范、所有领域知识一股脑写进 system prompt，demo 阶段没问题。但当能力涨到几十上百个，你会发现：

- 上下文窗口被说明书占满，留给真正任务的空间越来越少
- 每次请求都背着全部说明，token 成本线性上涨
- 一堆指令互相干扰，模型注意力被稀释，反而更容易出错

**Agent Skills 就是用来破这个局的。** 而它破局的核心机制，叫**渐进式披露（Progressive Disclosure）**。

这篇就讲清楚：Skill 是什么、渐进式披露好在哪，以及为什么它正好踩中 AI 工程的几根支柱。

---

## 一、Skill 是什么

一个 Skill，本质是一包**「按需加载的专业能力」**，通常由三部分组成：

```txt
描述(description)   —— 一句话说清「我能干啥、什么时候该用我」
指令(instructions)  —— 真正的操作知识 / SOP / 步骤
资源(可选)          —— 能直接执行的脚本、模板、参考文件
```

你可以把它理解成给 Agent 用的「**带说明书和工具的技能包**」。需要查飞书表格，就有一个飞书 Skill；需要做代码审查，就有一个 review Skill。

关键不在于"它是一包知识"，而在于**它是怎么被加载的**——这才是它和"直接写在 prompt 里"的本质区别。

---

## 二、核心机制：渐进式披露

渐进式披露的思想很简单：

> **不要一次把所有信息都摊给模型，而是分层、按需地给。**

具体到 Skill，是一个**三层加载**模型：

```txt
第 1 层：name + description
  → 始终在上下文里，但极短（一句话）
  → 模型靠它判断「这个任务要不要用这个 Skill」

第 2 层：完整 instructions（SKILL.md 正文）
  → 只有当模型判断「相关」时，才加载进来
  → 这里才是详细的操作步骤

第 3 层：捆绑的脚本 / 资源文件
  → 只有当这个 Skill 真正执行、需要某个文件时，才读取
```

用一张图对比就很直观：

```txt
传统做法（全量披露）：
  system prompt = 能力A说明 + 能力B说明 + ... + 能力Z说明
  → 100 个能力 = 100 份完整说明，全程常驻

渐进式披露：
  常驻的只有 100 句「一句话描述」
  完整说明？用到哪个才展开哪个
```

> 一句话：**能力可以有一百个，但上下文成本只跟"这次真正用到的那几个"挂钩。**

---

## 三、渐进式披露到底好在哪

这是最值得单独说清楚的部分。它的好处不止"省 token"一条：

### 1. 上下文窗口不被撑爆

上下文是有限资源。把说明书全塞进去，留给任务本身（用户输入、中间结果、历史）的空间就被挤占。渐进式披露让常驻内容保持精简，**把窗口留给真正干活的内容**。

### 2. 注意力不被稀释（哪怕窗口够大）

很多人以为"窗口够大就能随便塞"。但事实是：

> 上下文越长，模型越容易"找不到重点"——这就是常说的 needle-in-haystack 问题。

无关的说明越多，模型越容易被干扰、抓错重点。**少而相关 > 多而嘈杂。** 渐进式披露保证模型每次看到的，基本都是当前相关的。

### 3. 成本随"使用"而不是"拥有"增长

```txt
全量披露：成本 ∝ 你拥有多少能力
渐进披露：成本 ∝ 你这次用到多少能力
```

你可以挂 200 个 Skill，但某次任务只触发 2 个，就只为这 2 个付费。**能力库可以无限扩，成本不爆炸。**

### 4. 可扩展性：能力随便加

正因为成本和"拥有量"解耦，团队可以持续往里加 Skill，而不用担心"加一个就拖慢/拖贵所有请求"。这是它能**规模化**的根本。

### 5. 延迟更低

要处理的 token 少了，单次响应自然更快。

### 6. 减少指令冲突

一堆规范同时在场时，容易出现互相矛盾的指导。按需加载意味着**同一时刻在场的指令更少、更聚焦**，冲突概率下降。

> 小结这一节：渐进式披露 = **省钱 + 保注意力 + 可无限扩展 + 更快 + 更少冲突**。它不是单纯的优化技巧，而是让"能力规模化"成为可能的前提。

---

## 四、它正好踩中 AI 工程的几根支柱

如果你看过 AI 工程那套"五支柱"框架，会发现 Skill 几乎条条命中：

| AI 工程支柱 | Skill / 渐进式披露怎么帮 |
| --- | --- |
| **上下文（Context）** | 渐进披露，不撑爆窗口、不稀释注意力（最核心） |
| **成本（Cost）** | 不必每次背着全部指令，token 随使用增长 |
| **质量（Quality）** | 把最佳实践 / SOP 固化进去，行为一致、可复现 |
| **可靠性（Reliability）** | 能打包确定性脚本，该用代码的用代码，不靠模型即兴 |
| **可观测 / 可迭代（Observability）** | Skill 是文件，可版本管理、可 diff、改一处全员生效 |

---

## 五、为什么比"全塞 prompt / 微调 / RAG"好

- **vs 全塞 system prompt**：那个不可扩展，能力一多就互相干扰还贵；Skill 是"用时才取"。
- **vs 微调模型（fine-tune）**：微调慢、贵、黑盒、改不动；Skill **即写即生效、透明、可编辑**。
- **vs RAG**：RAG 擅长检索"事实知识"，Skill 擅长打包"怎么做的流程 + 可执行代码"。两者互补——RAG 给"是什么"，Skill 给"怎么做"。

---

## 六、最被低估的一点：确定性 + 灵活性的结合

Skill 可以把**指令和代码打包在一起**，这在 AI 工程里很关键：

```txt
确定性的活  → 交给 Skill 里的脚本（每次结果一样，可靠）
需要判断的活 → 交给模型读 Skill 指令后灵活处理
```

这正好回应 LLM"输入确定、输出不确定"的本质：

> **能用确定性代码兜住的部分，就别让模型概率性地猜。**

Skill 让你能优雅地划这条"代码 vs 模型"的分工线。

---

## 七、Skill 和 MCP：两种给 Agent 扩能力的思路

既然都是"给 Agent 加能力"，顺带和 MCP 对照一下（详见 [什么是 MCP](/2026/06/30/AI/what-is-mcp/)）：

```txt
MCP   ：标准化「Agent 怎么连接外部工具和数据源」（协议层、对外连接）
Skill ：打包「Agent 怎么做某件事」的知识、流程和脚本（能力层、行为指导）
```

简单说：

> **MCP 解决"接得上"，Skill 解决"会不会做"。** 一个偏连接，一个偏知识与流程，常常配合使用。

---

## 🧠 小结

```txt
1. Skill 是「按需加载的专业能力包」：描述 + 指令 + 可选脚本
2. 它的核心机制是渐进式披露：描述常驻、指令按需、资源用时才取
3. 渐进式披露的好处：省钱、不撑爆窗口、不稀释注意力、可无限扩展、更快、少冲突
4. 它踩中 AI 工程的上下文、成本、质量、可靠性、可观测多根支柱
5. 和 MCP 互补：MCP 管「接得上」，Skill 管「会不会做」
```

最关键的一句：

> Skill 的厉害之处不在"它装了多少知识"，而在"**它只在需要时才把知识拿出来**"——用渐进式披露，让能力可以无限扩展，而上下文成本只跟实际使用挂钩。这正是 AI 工程一直在追求的事。
